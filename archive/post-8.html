Thoughts on Nim.

Preface

Hey I'm back with another article. Let's talk about Nim.

History

According to Wikipedia: 

 Nim (formerly named Nimrod) is 
 an imperative, general-purpose, multi-paradigm, 
 statically typed, systems, compiled programming language

It was created by Andreas Rumpf. who I have mixed feelings about which I will touch later. this language compiles to C/C++/JavaScript among others.

Let's dive into Nim a bit.

NOTE: I did not use any features extensively that can't be sanely use without the Nim garbage collector (yes, Nim has a Garbage Collector, I'll get into that soon.)

Syntax

Ah, Syntax, My favorite part of language design.

If you do not know. I have mixed feelings about Pythonic Syntaxes. I think they're fine for small scripts, but for huge programs, it can be a mess just to find an indentation error. Nim tries to fix this by allowing you to also but explicit ()s, but that's largely discouraged by the community, and I'm pretty sure the compiler just ignores that.

In Nim, you define types, A LOT. heck the way you make structs is using the object keyword. which is (you guessed it), defined inside of a type Which isn;t really a bad thing, I think it's perfectly fine, but to some people it may be insanity. it uses camelCase, I prefer snake_case but whatever, I do not really care.

Useful Features

Defer

Defer Statements?!, really?, oh my god. defer in Go and Odin, or scope(exit) in D. is a god send, they're much nicer (in my opinion) than using RAII or try..finally clauses. when I discovered that Nim has these, I was immediately hooked. it's such a useful feature.

Seq

Sequences are an interesting take for Dynamic Arrays. especially since this language compiles to C. Regular "fixed" arrays are available too. Sadly if you want to avoid the garbage collector, you might not want to use this feature as it does need some GC things.

Macros

Macros in Nim are really nice, They feel more like Lisp/Rust Macros than the C preprocessor. I recommend clicking here for examples (C.W. OOP).

The Garbage Collector

The garbage collector in Nim has a few variations. For example, the stop-the-world, shared heap garbage collector similar to Go's garbage collector (Which is a very good garbage collector mind you, I barely have to think about it). however you can disable the garbage collector with the --gc:none option. Sadly this seems to be looked down upon by the community except for the some people who use Nim for embedded systems.

The Community

The community (for the most part) are nice, but they can be assertive on what you do. for example when I was asking how can I use Nim without garbage collection. I did not get any useful answers and instead was asked to use the experimental ARC "memory management strategy" by the community and Andreas Rumpf himself. it wasn't until exelotl actually gave an answer to my question.

In case if you're curious, this is the answer they gave. 

 Basically with gc:none you just wanna avoid all these features:
     - seq
     - ref / ref object
     - string  (use cstring instead)
     - closures / anonymous procedures 
     (if you wanna pass functions around, use {.nimcall.}, or {.nodecl.} if you wanna guarantee C compatibility)

 you might want to annotate all your object types with {.bycopy.} just for sanity 
 (apparently the compiler can choose to implicitly pass objects by reference if they get too big, though I've never actually experienced this)
 
 it's OK to use seqs and strings and closures in macros though
 just don't let them get into your runtime code

Conclusion

Overall, I think Nim has a lot of potential. I think the only thing I would change about it is its Pythonic Syntax and make it nicer to work with --gc:none. The community could be better but the ones that are close to me are nice people, so thank you exelotl!. This is IamRifki, signing off.
